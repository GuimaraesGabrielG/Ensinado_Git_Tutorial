Início no Projeto

Inicio no projeto

Para criar um ambiente local de controle de versão, é necessário inicializar o git no projeto:

$ git init

Após fazer alterações no projeto ou mesmo querer verificar o que mudou no seu projeto, usa - se o comando < git status > :

$ git status
On branch master

No commits yet

Untracked files:
(use "git add <file>..." to include in what will be committed)
arquivo.txt

Para adicionar o documento ou o projeto no git, pode ser feito de dois modos:

  • <git add "nome do arquivo" > . Adiciona arquivos selecionados .
  • <git add .>. Adiciona todos os arquivos.

Obs: O Documento só foi adicionado, ele não foi salvo.

 $ git add .
$ git status
On branch master

No commits yet

Changes to be committed:
(use "git rm --cached <file>..." to unstage)
new file:   arquivo.txt

O arquivo foi adicionado, para desfazer, você pode usar alinha <git rm --cached <"nome do arquivo">>

PARA SALVAR AS INFORMAÇÕES DEVEMOS COMMITAR:

  • Escrever um commit:
$ git commit -m "primeiro commit"
[master (root-commit) 3218596] primeiro commit
1 file changed, 2 insertions(+)
create mode 100644 arquivo.txt

Se fizer denovo o <git status>, o git vai responder que não tem nada para commitar.

Observações:

  • Esse processo deve feito várias vezes, ao longo do seu projeto, caso tenha alguma alteração, execute esses passos.
  • O git reconhece quando você muda algum conteúdo do seu arquivo.

Depois de fazer vários commits, é possivel ver todos os commit feitos, através do comando <git log>

$ git log

Date:   Tue Jul 14 21:56:12 2020 -0300

commit 0ede13030b31a63d77d61347cc9328c268bbaf8f

  • O git log, funciona como: mostrar o histórico de todos os commits feitos

     

INFORMAÇÕES SOBRE COMO COMMITAR:

  • Deixe a mensagem clara, muitos desenvolvedores utilizam o gerúndio, por exemplo, escrevendo, corrigindo, porém também pode ser feito no indicativo, o que deixa mais correto, por exemplo: "escreve", "adiciona", "corrige".
  • só escreva coisas relacionadas no que você fez no projeto, por exemplo, NUNCA coloque, "teste", "commit para fulano".
  • Quando corrigir um bug, EXPLIQUE o bug, nunca coloque, ""corrige o bug.", "fix bug".
  • Mantenha o padrão do histórico, se for ingles, sempre escreva inglês, se for gerundio, sempre escreva em gerundio.
